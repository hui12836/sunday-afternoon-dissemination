# 臉部平權網站專案：完整回顧與學習總結

> 寫給未來的自己，也寫給想用同樣方式做事的人。  
> 撰寫日期：2026-04-26

---

## 第一部分：這個專案是什麼

### 一句話說清楚

為陽光社會福利基金會製作臉部平權宣傳網站，目標是讓**從未聽過這個議題的一般人，在五分鐘內理解臉部平權是什麼、為什麼重要、自己能做什麼**。

### 基本資料

| 項目 | 內容 |
|------|------|
| 宣傳主題 | 臉部平權 × 5/17 臉部平權日 |
| 委託單位 | 陽光社會福利基金會 |
| 目標讀者 | 完全不知道這個議題的一般大眾 |
| 使用工具 | Perplexity（研究）+ Claude（設計與程式）+ Netlify（部署） |
| 開發方式 | 全程用自然語言指揮 AI，不自己寫程式 |
| 版本迭代 | v1 → v2 → v3.1，約 10 天完成 |
| 最終成果 | 可在手機和電腦正常瀏覽的單頁靜態網站 |

## 第二部分：AI 協作方法論（最核心的學習）

### 2-1 最重要的一個原則：把網站切成 6 塊，每次只給 AI 改一塊

這是整個專案最關鍵的設計決策。一開始不懂這件事，就很容易犯「請 AI 幫我把整個網站改好」的錯誤——這幾乎一定會出問題。

**6 個工作區塊的切法：**

| 編號 | 區塊名稱 | 包含什麼 |
|------|----------|----------|
| 01 | hero-and-nav | 導覽列、首屏大圖、主按鈕 |
| 02 | intro-and-principles | 「三要三不」等新手理解區 |
| 03 | six-cards-and-overlay | 六張議題卡片與點開後的詳情 |
| 04 | stories-video-links | 真實故事、影片專區、快速連結 |
| 05 | footer | 頁尾、社群、聯絡方式 |
| 06 | design-system | 全站 CSS、色彩、字級、無障礙規則 |

**下指令的正確方式：**

```
❌ 壞的指令：「請幫我把這個網站改好，讓它更清楚」

✅ 好的指令：
「這是我的 index.html。請只處理 04 影片區，需求如下：
1. 首頁只保留 1 支最具代表性的影片
2. 其他影片改成「更多影片」按鈕
3. 影片旁加入 1 句引導文
4. 保持其他區塊不變
請回傳完整 index.html 並列出你改了哪些地方。」
```

好指令的四個要素：**指定區塊 + 列出需求 + 明說不能動哪裡 + 要求回傳完整檔案並附說明**。

---

### 2-2 兩個 AI 的分工

這個專案用了兩個 AI，但它們做的事不一樣：

| AI | 角色 | 什麼時候用 |
|----|------|------------|
| **Perplexity** | 研究員 | 查台灣在地統計數據、找事實、驗證資訊、確認資料有來源 |
| **Claude** | 工程師 + 設計師 | 寫程式、改版面、調整文案結構、產出最終 HTML/CSS/JS |

**關鍵原則**：先用 Perplexity 把內容研究清楚，再把整理好的內容交給 Claude 實作。不要讓 Claude 同時做研究和寫程式——它可能會「幻想」出沒有來源的數據。

---

### 2-3 Token 管理：讓對話「不失憶」

**問題根源**：每次 Claude 收到新訊息，都會從頭重讀整個對話歷史。對話越長，token 消耗越快，AI 也越容易「遺忘」之前的細節。

**解決辦法：**

1. **把規格書改成 Markdown，不要用 HTML**  
   一份 700 行的 HTML 規格書 ≈ 8,000–10,000 token  
   精簡成 100 行的 Markdown 規格 ≈ 1,500 token（**省 85%**）

2. **建立兩份固定文件讓 AI 讀懂你的專案**  
   - `docs/change-brief.md`：這次改版目標與問題（告訴 AI 要做什麼）  
   - `docs/changelog.md`：每次修改紀錄（告訴 AI 做了什麼）  
   每次開新對話，把這兩份文件貼給 AI，它就不會「失憶」。

3. **每個任務開新對話**  
   不要在同一個對話裡改很多不同的事。每個獨立任務（改 footer、改卡片）開一個新對話，token 消耗少，AI 也更精準。

4. **截圖盡量裁切或改成文字**  
   一張 1000×1000 截圖 ≈ 1,300 token。能用文字描述就用文字。

## 第三部分：版本演變（v1 → v3.1）

| 版本 | 時間 | 改了什麼 | 為什麼要改 | 結果 |
|------|------|----------|------------|------|
| **v1** | 初始 | 單一 HTML 檔，所有 CSS/JS 內嵌 | 快速起步 | 可用，但結構混亂，AI 很難穩定修改 |
| **v2** | 中期 | 拆成 html/css/js/assets/docs 資料夾結構 | 讓 AI 每次只改一塊，降低出錯率 | 大幅提升穩定度，修改更有方向 |
| **v3.1** | 最終 | 重新定義色彩系統（暖色調）、完整內容資料庫、六大區塊重排、無障礙補強 | 設計審查後發現資訊架構不清楚 | 完成可部署版本，用於街頭宣傳 |

**每個版本切換的觸發點都是同一件事**：AI 指出「使用者看不懂」或「資訊優先序不清楚」。

## 第四部分：錯誤與教訓（最誠實的一章）

### 錯誤 1：寫了一份「不存在」的規格書

**發生了什麼**：花時間讓 AI 產出一份 700 行的 HTML 系統規格書（SRS），裡面包含報名表單、API 規格、時間軸議程。但實際網站根本沒有這些功能，兩者幾乎是兩個不同的網站。

**為什麼會這樣**：先寫規格書、再做網站，但過程中根據實際需求調整了很多，沒有回頭更新規格書。

**如何避免**：宣傳型的小型靜態網站不需要 SRS。一份 100 行的 Markdown 精簡規格就夠了，重點是**邊做邊更新，而不是一開始就寫完整**。

---

### 錯誤 2：圖片一直是佔位符

**發生了什麼**：首頁圖片用了 Unsplash 的泛用人物照，og:image 的 `{{OG_IMAGE_URL}}` 佔位符甚至直接部署上線——分享到 Facebook 或 LINE 時沒有任何預覽圖。

**為什麼會這樣**：一直在改功能和文案，圖片的問題被延後了又延後。

**如何避免**：把「換掉佔位符圖片」列進第一版 todo.md，**部署前先跑一遍搜尋 `{{` 確認沒有未填的佔位符**。

---

### 錯誤 3：無障礙問題在最後才發現

**發生了什麼**：viewport 設了 `maximum-scale=1.0`（阻止使用者縮放頁面）、部分文字對比度不足、按鈕缺少 `aria-label`——這些都是無障礙設計的基本問題，但都在最後的程式碼審查才被揪出來。

**特別諷刺的是**：這個網站的主題是臉部平權，倡議的是包容與尊重，網站自己卻在無障礙上出了問題。

**如何避免**：在第一版就把無障礙清單加進 change-brief，不要等到最後才補。至少要確認：`alt` 文字、`aria-label`、鍵盤操作、顏色對比度。

---

### 錯誤 4：資料沒有標來源

**發生了什麼**：網站上的統計數字（「每五人就有一人受到不友善對待」）沒有標明出自陽光基金會 2025 年的調查報告。被 AI 審查指出後才補。

**為什麼會這樣**：讓 AI 寫內容時，沒有要求它每個數據都附來源。

**如何避免**：叫 Perplexity 查數據時，**要求它列出資料來源的機構名稱和年份**。寫進網站後記得一起放上「資料來源」標示。

---

### 錯誤 5：一開始讓 AI 一次改整站

**發生了什麼**：最初幾次修改，是把整個 HTML 丟給 AI 說「幫我改好」。結果 AI 一次改了太多地方，某些區塊被弄壞或邏輯亂掉，追蹤問題很困難。

**如何避免**：這就是切成 6 個區塊的原因（見第二部分 2-1）。

---

### 如果重來，我會這樣做

1. **第一天就建立** `docs/change-brief.md` 和 `docs/changelog.md`
2. **第一版就補無障礙清單**（alt、aria、鍵盤、對比度）
3. **搜尋 `{{` 才部署**，確認沒有未填的佔位符
4. **每個數據要求 Perplexity 附來源**
5. **每次改版後，先在手機上看一遍**，確認沒有跑版
6. **圖片換成真實素材再部署**，不要「之後再說」

## 第五部分：內容研究方法（如何讓文案真的讓人想讀）

### 5-1 研究流程

```
Perplexity 查 → 整理成 Markdown → 交給 Claude 轉換成網站文案
```

Perplexity 的強項是找**台灣在地的最新統計數據**（例如陽光基金會 2025 年問卷調查結果）。Claude 的強項是把數據和故事整合成有節奏的文案。

---

### 5-2 讓人想讀下去的寫法

這個專案學到了一個很有用的內容寫作原則，來自分析國際特赦組織（Amnesty International）的網站：

> **先講人，不是先講制度。**

**範例對比：**

```
❌ 一般寫法：
「臉部平權是指不因顏面外觀歧視任何人，陽光基金會自 2010 年起推動此運動。」

✅ 讓人想讀的寫法：
「想像你走在街上，一場意外燒傷了你的臉。
從那天起，陌生人盯著你看，老闆在面試時猶豫不決。
你沒變笨，沒變懶，只是臉變了樣，卻開始被當成「異類」。」
```

**三個關鍵技巧：**
1. **從人的具體處境切入**，不要從定義切入
2. **只揭露一部分問題**，用懸念讓讀者想繼續看
3. **帶有情緒重量**：用「隱形的牢籠」「比疤痕更深的痛」這類具象的說法

---

### 5-3 六大主題的排列順序很重要

最終採用的順序是：「是什麼 → 有哪些誤解 → 當事人會遇到什麼 → 為什麼需要這個運動 → 你能做什麼 → 如何參與」

這是**新手閱讀路徑**：先給定義，再給同理，最後給行動。如果順序顛倒，讀者還沒建立基礎認知就被要求行動，很容易直接離開。

## 第六部分：給未來自己的備忘錄

### 開始任何類似專案的前 10 個動作

1. 建立 `docs/change-brief.md`（目標、問題、修改方向）
2. 建立 `docs/changelog.md`（記錄每次改了什麼）
3. 建立 `docs/todo.md`（還沒做的事）
4. 把網站切成 5-6 個區塊，每個取個名字
5. 決定哪個 AI 做研究（Perplexity），哪個 AI 做實作（Claude）
6. 讓 Perplexity 把所有統計數據列出來並附來源
7. 把規格寫成 100 行以內的 Markdown，不要用 HTML
8. 先確認手機版沒問題，再確認桌機版
9. 部署前搜尋 `{{` 確認沒有佔位符
10. 每個區塊改完就部署一次，確認沒有壞版

### 什麼時候開新對話

- 換了一個新的工作區塊（例如從改 footer 變成改六張卡片）
- 對話已經超過 20 幾則
- AI 開始出現奇怪的錯誤或忘記之前的決定
- 任何新的獨立任務

## 第七部分：給其他人的入門指南

> **前提：你完全不需要會寫程式。你需要的是「會指揮 AI」。**

### 三步驟啟動流程

**第一步：用 Perplexity 做研究**

問題模板：
```
我想做一個關於 [你的主題] 的宣傳網頁，目標讀者是 [誰]。
請幫我找出：
1. 台灣（或你的地區）最相關的統計數據（附來源）
2. 大眾最常有的誤解
3. 6–8 個適合作為網頁主題的問題式標題
```

**第二步：用 Claude 建立第一版**

問題模板：
```
我要做一個宣傳 [主題] 的單頁靜態網站。
目標讀者是從未聽過這個議題的一般人，希望他們 5 分鐘內能理解。

網站需要以下區塊：
- 首屏（標題 + 一句說明 + 按鈕）
- [列出你的 3-6 個主題區塊]
- 頁尾

請用現代、有溫度、易讀的風格設計，輸出完整可用的 HTML 單頁網站。
```

**第三步：逐塊修改**

每次只改一塊，用這個模板：
```
這是我的 index.html。
請只修改「[區塊名稱]」這個部分，需求如下：
1. [具體需求 1]
2. [具體需求 2]
不要改動其他區塊。
請回傳完整 index.html 並列出你改了哪些行。
```

---

### 常見問題與解法

| 問題 | 解法 |
|------|------|
| AI 改了不該改的地方 | 在指令裡加「不要碰 [區塊名稱]」 |
| AI 忘記之前的設定 | 開新對話，把 change-brief.md 貼進去 |
| 不知道哪裡壞了 | 問 AI「請比對我之前的版本和現在的版本，列出你改了哪些地方」 |
| 想讓設計更好看 | 先找一個參考網站（如 Amnesty Taiwan），告訴 AI「請參考這個網站的排版風格」 |
| Token 消耗太快 | 開新對話 + 把規格簡化成 Markdown + 截圖改成文字描述 |
| 資料不知道可不可信 | 要求 Perplexity 附上機構名稱和年份，不要直接用 AI 自己說的數字 |
