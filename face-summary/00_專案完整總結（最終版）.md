## 第一部分：這個專案是什麼

### 一句話說清楚

為陽光社會福利基金會製作臉部平權宣傳網站，目標是讓**從未聽過這個議題的一般人，在五分鐘內理解臉部平權是什麼、為什麼重要、自己能做什麼**。

### 基本資料

| 項目 | 內容 |
|------|------|
| 宣傳主題 | 臉部平權 × 5/17 臉部平權日 |
| 委託單位 | 陽光社會福利基金會（非官方，學生自製宣導） |
| 目標讀者 | 完全不知道這個議題的一般大眾 |
| 使用工具 | Perplexity（研究） + Claude（設計與程式） + Netlify（部署） |
| 開發方式 | 全程用自然語言指揮 AI，不自己寫程式 |
| 版本迭代 | v1 → v2 → v3.1，約 10 天完成 |
| 最終成果 | 可在手機和電腦正常瀏覽的單頁靜態網站 |

### 網站最終架構（從上到下）

| 區塊 | 內容 |
|------|------|
| Hero 首屏 | 標題「每張臉都與眾不同，卻一樣獨一無二」+ 5/17 標籤 |
| 三要三不 | 互動原則：平常心、自然互動、正面鼓勵 / 不盯看、不問隱私、不指點 |
| 八張議題卡片 | 點開後有完整 overlay 詳細內文（仿 Amnesty Taiwan 風格） |
| 真實故事 | 職場歧視案例、采瑀、沈曉亞、小芳、阿蕎（附連結） |
| 影響力數據 | 45年、9000+倖存者、1700人/年、16據點 |
| 影音專區 | 5 部 YouTube 影片（含縮圖預覽、點擊直跳） |
| 快速連結 | 臉部平權、職場指引、影片、故事、IG、FB |
| 頁尾 | 社群 + 聯絡資訊 + 免責聲明（收合式設計） |

## 第二部分：研究流程（如何讓內容有根據）

### 兩個 AI 的分工

| AI | 角色 | 用在哪裡 |
|----|------|----------|
| **Perplexity** | 研究員 | 查台灣在地統計、找事實、驗證數據、確認資料來源 |
| **Claude** | 工程師 + 設計師 | 寫程式、調版面、整合文案、產出最終 HTML/CSS/JS |

**關鍵原則**：先用 Perplexity 把內容研究清楚（含來源），再把整理好的內容交給 Claude 實作。讓 Claude 同時查數據和寫程式，它可能會「幻想」出沒有根據的數字。

### 研究問題模板（可直接複製使用）

```
我想做一個關於 [你的主題] 的宣傳網頁，目標讀者是 [誰]。
請幫我找出：
1. 台灣最相關的統計數據（附機構名稱與年份）
2. 大眾最常有的誤解
3. 6–8 個適合作為網頁主題的問題式標題
```

### 讓文案讓人想讀的核心技巧

**原則：先講人的處境，不是先講制度。**（來自分析 Amnesty Taiwan 的設計）

```
❌ 一般寫法：
「臉部平權是指不因顏面外觀歧視任何人，陽光基金會自 2010 年起推動此運動。」

✅ 讓人想讀的寫法：
「想像你走在街上，一場意外燒傷了你的臉。
從那天起，陌生人盯著你看，老闆在面試時猶豫不決。
你沒變笨，沒變懶，只是臉變了樣，卻開始被當成「異類」。」
```

**三個關鍵技巧：**
1. 從人的具體處境切入，不要從定義切入
2. 只揭露一部分問題，用懸念讓讀者想繼續看
3. 帶有情緒重量：用「隱形的牢籠」「比疤痕更深的痛」這類具象說法

### 六到八個主題的排列順序很重要

最終採用的順序：**是什麼 → 有哪些誤解 → 當事人遇到什麼 → 為什麼需要這個運動 → 你能做什麼 → 成功故事 → 如何參與**

這是**新手閱讀路徑**：先給定義，再給同理，最後給行動。順序顛倒，讀者還沒建立認知就被要求行動，很容易離開。

## 第三部分：AI 協作方法論（最核心）

### 3-1 把網站切成 6 塊，每次只給 AI 改一塊

| 編號 | 區塊名稱 | 包含什麼 |
|------|----------|----------|
| 01 | hero-and-nav | 導覽列、首屏大圖、主按鈕 |
| 02 | intro-and-principles | 「三要三不」等新手理解區 |
| 03 | six-cards-and-overlay | 六張議題卡片與點開後的詳情 |
| 04 | stories-video-links | 真實故事、影片專區、快速連結 |
| 05 | footer | 頁尾、社群、聯絡方式 |
| 06 | design-system | 全站 CSS、色彩、字級、無障礙規則 |

**下指令的正確方式：**

```
❌ 壞的指令：「請幫我把這個網站改好，讓它更清楚」

✅ 好的指令：
「這是我的 index.html。請只處理 04 影片區，需求如下：
1. 首頁只保留 1 支最具代表性的影片
2. 其他影片改成「更多影片」按鈕
3. 影片旁加入 1 句引導文
4. 保持其他區塊不變
請回傳完整 index.html 並列出你改了哪些地方。」
```

好指令的四要素：**指定區塊 + 列出需求 + 明說不能動哪裡 + 要求回傳完整檔案並附說明**

### 3-2 Token 管理：讓對話不「失憶」

**問題根源**：每次 Claude 收到新訊息，都會從頭重讀整個對話歷史。對話越長，AI 越容易遺忘之前的細節。

**四個解決辦法：**

| 方法 | 節省量 | 做法 |
|------|--------|------|
| 把規格書改成 Markdown | 省 85% token | 700 行 HTML ≈ 10,000 token；100 行 Markdown ≈ 1,500 token |
| 建立兩份固定文件 | 讓 AI 快速進入狀態 | `change-brief.md`（要做什麼） + `changelog.md`（做了什麼） |
| 每個任務開新對話 | 避免累積誤差 | 改 footer 開一個、改卡片開一個 |
| 截圖改成文字描述 | 省 1,300 token/張 | 能文字描述的就不要傳截圖 |

### 3-3 什麼時候開新對話

- 換了一個新的工作區塊
- 對話已超過 20 則
- AI 開始出現奇怪錯誤或忘記之前的決定
- 任何新的獨立任務

## 第四部分：版本演變（v1 → v3.1）

| 版本 | 時間 | 改了什麼 | 為什麼要改 | 結果 |
|------|------|----------|------------|------|
| **v1** | 初始 | 單一 HTML 檔，所有 CSS/JS 內嵌，六張卡片基礎版 | 快速起步 | 可用，但結構混亂，AI 很難穩定修改 |
| **v2** | 中期 | 拆成 html/css/js/assets/docs 資料夾結構；加入真實圖片與 YouTube 影片 | 讓 AI 每次只改一塊，降低出錯率 | 大幅提升穩定度，修改更有方向 |
| **v3.1** | 最終 | 重新定義色彩系統（暖色調）、完整六大 overlay 詳細頁面、無障礙補強、Code Review 12 項修正 | 設計審查後發現資訊架構不清楚 | 完成可部署版本，用於街頭宣傳 |

**每個版本切換的觸發點**：AI 指出「使用者看不懂」或「資訊優先序不清楚」。

## 第五部分：錯誤與教訓（最誠實的一章）

### 錯誤 1：寫了一份「不存在」的規格書

**發生了什麼**：花時間讓 AI 產出一份 700 行的 HTML 系統規格書，裡面包含報名表單、API 規格、時間軸。但實際網站根本沒有這些功能。

**教訓**：宣傳型靜態網站不需要 SRS。一份 100 行的 Markdown 精簡規格就夠，重點是**邊做邊更新**。

---

### 錯誤 2：圖片一直是佔位符

**發生了什麼**：`og:image` 的 `{{OG_IMAGE_URL}}` 佔位符直接部署上線——分享到 Facebook 或 LINE 時沒有預覽圖。

**教訓**：部署前執行搜尋 `{{`，確認沒有未填的佔位符。把「換掉佔位符圖片」列進第一版 todo.md。

---

### 錯誤 3：無障礙問題在最後才發現

**發生了什麼**：`viewport` 設了 `maximum-scale=1.0`（阻止使用者縮放）、按鈕缺少 `aria-label`、部分文字對比度不足——這些都在最後的 Code Review 才被揪出來。

**特別諷刺**：網站主題是臉部平權，倡議包容與尊重，自己卻在無障礙上出問題。

**教訓**：第一版就把無障礙清單加進 change-brief。至少要確認：`alt` 文字、`aria-label`、鍵盤操作、顏色對比度。

---

### 錯誤 4：數據沒有標來源

**發生了什麼**：網站上的統計數字沒有標明來自哪份調查。被 AI 審查指出後才補。

**教訓**：叫 Perplexity 查數據時，要求它每個數字都附機構名稱和年份。寫進網站後記得一起放上「資料來源」標示。

---

### 錯誤 5：一開始讓 AI 一次改整站

**發生了什麼**：最初把整個 HTML 丟給 AI 說「幫我改好」。結果 AI 一次改了太多地方，某些區塊被弄壞，追蹤問題很困難。

**教訓**：切成 6 個區塊，每次只改一塊（見第三部分 3-1）。

---

### 如果重來，我會這樣做

1. **第一天就建立** `docs/change-brief.md` 和 `docs/changelog.md`
2. **第一版就補無障礙清單**（alt、aria、鍵盤、對比度）
3. **搜尋 `{{` 才部署**，確認沒有未填的佔位符
4. **每個數據要求 Perplexity 附來源**
5. **每次改版後，先在手機上看一遍**，確認沒有跑版
6. **圖片換成真實素材再部署**，不要「之後再說」

## 第六部分：設計參考來源（Amnesty Taiwan 的啟示）

這個專案的設計靈感來自 https://www.amnesty.tw/issues 的深度分析，最值得學的三點：

| 設計原則 | Amnesty 怎麼做 | 這個專案如何應用 |
|----------|----------------|-----------------|
| **低摩擦導覽** | 每個議題區塊遵循相同模板（標題→導言→了解更多） | 八張卡片採用一致模板，降低認知負擔 |
| **先講人** | 引言直接描述人的處境（「很多人因行使自由而遭受監禁」） | 每張 overlay 開頭都從具體場景切入 |
| **一屏一個核心畫面** | 每段只留一個核心概念，短、硬、清楚 | 卡片摘要控制在一到兩句，細節進 overlay |

## 第八部分：給其他人的三步驟啟動流程

> **前提：你完全不需要會寫程式。你需要的是「會指揮 AI」。**

### 第一步：用 Perplexity 做研究

```
我想做一個關於 [你的主題] 的宣傳網頁，目標讀者是 [誰]。
請幫我找出：
1. 台灣最相關的統計數據（附來源：機構名稱 + 年份）
2. 大眾最常有的誤解
3. 6–8 個適合作為網頁主題的問題式標題
```

### 第二步：用 Claude 建立第一版

```
我要做一個宣傳 [主題] 的單頁靜態網站。
目標讀者是從未聽過這個議題的一般人，希望他們 5 分鐘內能理解。

網站需要以下區塊：
- 首屏（標題 + 一句說明 + 按鈕）
- [列出你的 3-6 個主題區塊]
- 頁尾

請用現代、有溫度、易讀的風格設計，輸出完整可用的 HTML 單頁網站。
```

### 第三步：逐塊修改

```

請回傳完整 index.html 並列出你改了哪些行。
```

## 第九部分：常見問題速查

| 問題 | 解法 |
|------|------|
| AI 改了不該改的地方 | 在指令裡加「不要碰 [區塊名稱]」 |
| AI 忘記之前的設定 | 開新對話，把 change-brief.md 貼進去 |
| 不知道哪裡壞了 | 問 AI「請比對我之前的版本和現在的版本，列出你改了哪些地方」 |
| 想讓設計更好看 | 先找一個參考網站，告訴 AI「請參考這個網站的排版風格」 |
| Token 消耗太快 | 開新對話 + 把規格簡化成 Markdown + 截圖改成文字描述 |
| 資料不知道可不可信 | 要求 Perplexity 附機構名稱和年份，不要直接用 AI 自己說的數字 |
| 部署後連結壞了 | 搜尋 `{{` 確認沒有佔位符；所有超連結用 `www.` 主站避免子網域問題 |
| 網站不會消失嗎？ | Netlify Free Plan 永久免費，只要不超過 100GB/月頻寬就不會下線 |
